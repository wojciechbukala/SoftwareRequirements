Implement a program that simulates a seat reservation system for an event management company. The program must manage events, seats, and reservations through a web application accessible from modern web browsers and enforce all pricing, availability, and booking rules described below. All users access the system through a single Login / Register panel, which routes them after authentication to either the Customer interface or the Administrator interface based on their role.

Each event has a unique identifier, a scheduled date, and a seating grid defined by the fixed number of rows and columns. All seats within a single event are either numbered or non-numbered. Every seat has a category: Economy, Premium or VIP. 

The price for each seat class at each event is calculated dynamically. An Early Bird discount of 20% is applied to reservations made more than 30 days before the event. A Last Minute surcharge of 30% is applied if the reservation is made less than 48 hours before the event and more than 80% of the seats in that event are already sold; "sold" here refers to seats that are either currently held by a pending reservation or already confirmed. 

The base prices set by the Administrator must satisfy a strict hierarchy: the base price of a VIP seat shall be greater than that of a Premium seat, and the base price of a Premium seat shall be greater than that of an Economy seat. After applying Early Bird or Last Minute modifiers, the effective price of a VIP seat shall remain greater than or equal to that of a Premium seat, and the effective price of a Premium seat shall remain greater than or equal to that of an Economy seat.

User has 15 minutes to confirm payment after locking seats, otherwise seats return to the available pool. User can hold maxium of 2 pending reservations at any given time. Each reservation is atomic, so either all requested seats are successfully reserved, or none of them are. For numbered events, all seats within a single reservation must be adjacent, to avoid single unsold places. Adjacency is defined as a contiguous block of seats within a single row (horizontal adjacency only).

If payment is not confirmed within 15 minutes, the reservation transitions to the Expired state and seats return to the pool; this is distinct from user-initiated cancellation of a confirmed reservation.

A confirmed reservation may be canceled (by its owner) at any time except 2 hours befeore the event. This way all seats from reservation retures to the pool. 

The administrator interface must allow creating and configuring events, defining seating grides and categories, setting base prices, and viewing reservation summaries per event. The user interface must allow browsing available events, selecting andreserving seats, completing or abandoning payment, and cancelling confirmed reservations. The visual design of the Login, User, and Administrator interfaces shall follow the mockups provided in the UI_Mockups folder (LoginRegister.png, User.png, Admin.png); views not covered by mockups shall follow the same graphical style.
