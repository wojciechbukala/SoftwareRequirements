Implement a program that simulates the onboard computer (named Copilot) of a driver-assistance system. The program reads sensor readings and driver events from CSV input files, processes them, and writes the results to the CSV output files.

All input and output files use CSV format with a comma as the field separator and UTF-8 encoding. Input consists of two files: *sensor_log.csv* — periodic readings from on-board camera and Lidar sensors, where each row represents one sensor event that triggers a Copilot processing cycle and includes columns such as timestamp, sensor_id, sensor_type, data_value, unit; and *driver_events.csv* — events generated by the driver, including timestamp, event_type, value. Events from both files are processed in a single stream ordered by ascending timestamp.

Output should consists of three files: *state_log.csv* — every system state transition with its timestamp, previous_state, current_state, trigger_event; *commands_log.csv* — every command issued to an actuator, containing a timestamp, actuator_id, values; and *feature_decision.csv* — records a timestamp alongside the autonomous feature involved — one of cruise control, emergency braking, or lane keeping — and the decision made for that feature (e.g. a correction value or brake/no brake), with column names timestamp, feature, decision .

When the program is in disengaged mode, it logs sensor data and does not issue actuator commands for lane keeping or cruise control; emergency braking is the sole exception and remains active in every mode. When the driver switches to engaged mode, the program processes each sensor event and issues commands to actuators for lane keeping, cruise control, and emergency braking. Emergency braking always has the highest priority and is evaluated on every Lidar reading regardless of the current mode. A Lidar reading with a distance strictly less than 5 m triggers emergency braking: the system issues a braking command to the Braking System actuator and records a BRAKE decision, whether the program is engaged or disengaged.

While engaged, Copilot periodically prompts the driver for an attentiveness check by issuing a small steering wheel movement command every 120 seconds, then waits up to 5 seconds for a driver response. Three outcomes are possible:
1) If a steering wheel force arrives within the required time and its value is 3 N or less, the system remains engaged and continues normally. The 120-second attentiveness interval is restarted from this moment, so the next prompt is issued 120 seconds after the valid response.
2) If no steering wheel force arrives within 5 seconds, the system emits an alarm and maintains it until the driver responds.
3) If a steering wheel force arrives within the required time but its value is above 3 N and below 10 N, the signal is ignored and the system continues waiting for a valid response.
A steering wheel force strictly greater than 10 N causes immediate disengagement of Copilot at any time, regardless of the current state.

A state transition is an actual change of state; events that keep the system in the same state shall not be recorded in *state_log.csv*.

The program is invoked from the command line as copilot --input <dir> --output <dir>, where both arguments are mandatory and specify, respectively, the directory containing the input files and the directory where output files are written.
